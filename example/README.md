# react-native-playback-controls — example

A demo app that exercises the whole library API on both platforms: starting a
system "now playing" session, updating metadata/playback state as a fake
player runs, handling remote commands, and tearing the session down.

## Why there's audio in here

iOS only shows lock-screen / Control Center controls for the app that is
actually rendering audio through an active `AVAudioSession`. The library
itself (by design) never touches the audio session — it only drives
`MPNowPlayingInfoCenter` / `MPRemoteCommandCenter` on iOS and a media3
`MediaSessionService` on Android. So this example app loops a ~2s silent WAV
file (`assets/silence.wav`) via `expo-audio` whenever the fake player's status
is `'playing'`, purely to satisfy iOS's audio-session requirement. Android
doesn't need this — the media notification appears regardless — but the app
does it on both platforms for consistency.

The playlist, position, transport state, etc. are **entirely simulated** in JS
(`src/useFakePlayer.ts`); no real media is decoded or streamed.

## Running

From the repo root:

```sh
yarn                                   # install workspace deps
cd example && npx expo prebuild --clean  # regenerate ios/ and android/ (gitignored)
yarn example ios                       # or: yarn example android
```

`example/ios` and `example/android` are generated by `expo prebuild` and are
not committed — regenerate them after pulling changes that touch
`example/app.json`, `example/package.json`, or the library's native/codegen
config.

## What to look for

1. Tap **Start session**. On Android 13+ you'll first be asked for the
   notification permission — the app requests it explicitly before starting
   the session (the library deliberately never requests runtime permissions
   itself).
2. **iOS**: background the app (or lock the device — a physical device or a
   simulator with the hardware "Lock" works) and check the Control Center /
   lock screen "Now Playing" card. Title, artist, artwork, position, and the
   transport buttons should reflect the fake player. Press play/pause, skip
   ±15s, next/previous track, and scrub the progress bar; each should show up
   in the on-screen event log. A physical device (or `AirPods`/Bluetooth
   remote via a simulator's paired device) is needed to exercise hardware
   media-button presses — the Control Center UI itself works fine in the
   simulator.
3. **Android**: check the system notification shade — a media-style
   notification should appear with the app's icon, track metadata, and
   transport buttons, plus the same card on the lock screen. Pressing a
   notification button should update the event log within the app.
4. **Bluetooth**: play/pause/skip buttons on a paired Bluetooth headset or car
   head unit route through the same remote-command path as the on-screen
   system controls — no extra wiring needed.
5. The **event log** (bottom of the screen, last 20 entries) shows every
   remote command the OS delivered, with a timestamp — use it to confirm a
   press actually round-tripped through native code back into JS.
6. **End session** tears the system controls down immediately; the session is
   also ended automatically if the app unmounts.

## Structure

- `src/playlist.ts` — hardcoded 3-track demo playlist (one track's artwork is
  a local `require()` asset, the other two are remote URLs).
- `src/useFakePlayer.ts` — the fake player: a `useReducer`-based state machine
  for status/position/rate/track, a 1s UI-only position ticker, and the
  silent `expo-audio` loop.
- `src/useEventLog.ts` — keeps the last 20 timestamped remote-command events
  for on-screen display.
- `src/App.tsx` — wires the fake player to the library: starts/ends the
  session, pushes `setNowPlaying` / `setPlaybackState` on every fake-player
  transition, and handles commands via both `useRemoteCommand` (for `play`
  and `pause`, to demonstrate the hook) and a catch-all
  `session.addCommandListener` (for everything else, plus logging).
