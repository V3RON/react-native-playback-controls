## README structure (required sections)

1. **Header** — one-paragraph pitch: control the system media UI (lock screen,
   Control Center, Android media notification, Bluetooth buttons) without
   playing audio; modern replacement for `react-native-music-control`;
   **New Architecture only**.
2. **Requirements** — RN version floor (New Arch, codegen EventEmitter ⇒ state
   the tested floor honestly: developed against RN 0.83), iOS & Android
   minimums (check `PlaybackControls.podspec` / `android/build.gradle` for the
   real values).
3. **Installation** — yarn/npm + pod install; note: no old-architecture
   support, no extra native setup beyond the platform sections below.
4. **iOS setup** (critical, most support issues live here):
   - Controls appear only while the app plays audio through an active
     `AVAudioSession` (`.playback`). This library deliberately never touches
     the audio session — your player (react-native-video, expo-audio, native
     AVPlayer…) owns it.
   - `UIBackgroundModes: audio` in Info.plist for background command delivery.
5. **Android setup**:
   - Permissions the library's manifest already merges vs. what the app must
     do itself: request `POST_NOTIFICATIONS` at runtime on Android 13+.
   - `startSession` must be called while the app is in the foreground.
   - Controls are dismissed when the user swipes the app away (by design —
     dead JS cannot handle presses).
6. **Quick start** — the canonical example (session start, metadata, state
   snapshot, command listener, cleanup). Copy the shape from `example/src` and
   keep it under ~40 lines.
7. **API reference** — generated by hand from the exported JSDoc:
   `PlaybackControls.startSession` / `isSessionActive`, `PlaybackSession`
   methods, `useRemoteCommand`, and every public type with field tables
   (defaults, units, platform notes).
8. **Platform behavior notes** — an honest matrix:
   - `toggle-play-pause`: iOS-only as an event; Android always delivers
     `play`/`pause` (register all three).
   - Position is snapshot-based: call `setPlaybackState` on transitions/seeks
     only; both platforms interpolate the seek bar. Never poll.
   - Button placement/ordering is OS-controlled on both platforms.
   - `change-playback-rate` surfaces differently (iOS lock screen vs Android).
   - Exactly one thing in the app may own the media session — conflicts with
     react-native-video's `showNotificationControls`, react-native-track-player
     etc.
9. **Troubleshooting** — at minimum: "controls don't show on iOS" (no active
   audio session / not the Now Playing app), "no notification on Android 13+"
   (POST_NOTIFICATIONS), "`session-already-active`", "`foreground-required`",
   "controls disappear after app swipe" (by design), "buttons do nothing after
   Metro reload" (should not happen — `invalidate` handles it; if reproduced,
   file it).
10. **Roadmap** — link `docs/IDEAS.md`. **Contributing/License** — keep the
    existing links.
